# AI 需求反向工程

## 核心理念

Vibe Coding 没有常识，只有逻辑。如果输入端是模糊的，输出端就是灾难。

这个 skill 解决的问题：**需求细节规则的确定最耗时**。传统做法是人去填补空白，这里反过来 — 让 AI 挖掘空白，让产品经理做选择题而非写小作文。

## 工作流概览

三步分步执行，每步独立产出，用户可以在任意步骤暂停，带着产品反馈回来后继续下一步。

```
Step 1: 防御性需求清单 → 输出选择题/判断题给产品
Step 2: 决策矩阵       → 将沟通结论转为结构化表格
Step 3: 策略模式骨架    → 生成可扩展的 Spring Boot 代码框架
```

用户可以选择从任意步骤开始。如果用户只说"分析这个需求"，默认从 Step 1 开始。

---

## Step 1: 防御性需求清单

### 目标
扮演资深业务架构师 + 严苛 QA 工程师，从产品给出的需求框架中挖出**真正缺失的**细节，输出**选择题/判断题**形式的问题清单。

### 输入
用户粘贴的初步需求框架（文字、截图、文档链接均可）。

### 执行流程

1. **上下文摸底（关键前置步骤）**

   在生成任何问题之前，先向用户确认已有背景：

   > "在我生成问题清单之前，想先了解一下：
   > 1. 这个模块属于哪个更大的系统？是否有已定义好的数据模型/表结构/接口？
   > 2. 有哪些业务规则是你已经跟产品确认过的？（直接告诉我，我会跳过这些）
   > 3. 这次需求的核心关注点是什么？（是从零搭建，还是在已有系统上新增功能？）"

   如果用户已经在对话中提供了上下文（如"这是学业评价系统的评价跟进模块"），从中提取已知信息，不要再追问已明确的部分。

   **原则：只问你不知道的、产品也可能没想到的。**

2. **阅读需求框架**，提取核心业务对象、流程、角色。

3. **基于已知上下文过滤**，将六个扫描维度中"已有答案"的部分标记为已知，只针对**信息缺口**生成问题。

4. **从六个维度扫描缺失细节**（跳过已有答案的部分）：

   对每个维度，先判断用户提供的上下文中是否已有明确答案。如果某个点的答案已经清楚，不要再生成问题。只针对**确实模糊或缺失**的细节提问。

   **a) 状态流转**
   - 列出所有可能的状态机流转路径（包括正向、逆向、异常分支）
   - 追问：每个异常状态如何处理？是否可回退？回退后数据怎么办？

   **b) 边界条件（Edge Cases）**
   - 至少列举 5 个极端场景，重点关注：
     - 并发竞争（两人同时操作同一资源）
     - 超时处理（流程中断后如何恢复）
     - 数据为空/为 null（关键字段缺失时的行为）
     - 幂等性（重复提交是否安全）
     - 数值溢出/精度（金额计算、数量上限）

   **c) 角色与权限**
   - 梳理所有涉及的角色
   - 追问越权操作的阻断规则（谁能看、谁能改、谁能删）

   **d) 数据一致性**
   - 跨表/跨服务的数据同步策略
   - 分布式事务边界

   **e) 外部依赖**
   - 第三方接口的失败降级方案
   - 回调/通知的重试与兜底

   **f) 业务规则冲突**
   - 多条规则同时满足时的优先级
   - 规则互斥时的处理策略

5. **输出格式** — 所有问题必须以**选择题或判断题**形式呈现，让产品经理能快速作答。

   **数量控制**：问题总数控制在 **10-25 个**之间。如果用户提供了较多上下文，问题数应更少（聚焦真正的盲区）。宁可少而精，不要多而泛 — 产品经理收到 40 个问题会直接忽略。

   **已知信息处理**：如果某个维度的细节已从用户上下文中得知，在该维度开头用一行简述已知结论，然后只追问衍生的未知问题。例如：
   > ✅ 已知：评价数据的来源分为"备课组创建"和"个人创建"两种。
   > 以下仅针对未明确的部分提问：

```markdown
## 防御性需求清单

### 状态流转
Q1. [选择题] 当订单处于"已支付"状态时，用户发起退款，订单状态应变为？
   A. 直接变为"已退款"
   B. 先变为"退款中"，等财务审批后再变为"已退款"
   C. 其他：___

Q2. [判断题] 订单取消后，已发放的优惠券是否需要自动回收？ 是 / 否

### 边界条件
Q3. [选择题] 用户在支付过程中网络超时，但支付已实际扣款成功，系统应？
   A. 依赖支付回调自动补单
   B. 用户手动查询后补单
   C. 同时支持 A 和 B
...
```

4. **产出清单后**，告诉用户："拿着这份清单去找产品经理，让他们做选择题。回来后把答案给我，我帮你生成决策矩阵。"

## Step 2: 决策矩阵

### 目标
将产品经理的答案 + 口头结论转化为 Markdown 格式的**业务规则决策表**，这是 Vibe Coding 时最高效的 Prompt 输入格式。

### 输入
以下任意一种：
- Step 1 的问题清单 + 产品经理的答案
- 会议纪要 / 聊天记录要点
- 用户口述的业务规则补充

### 执行流程

1. **解析输入**，提取所有已确定的业务规则。

2. **生成决策矩阵**，模板见 `references/decision-matrix-template.md`。表头固定为：

| 场景条件 | 触发动作 | 异常抛出 | 对应状态变更 | 备注 |
|---------|---------|---------|------------|-----|

3. **补充衍生规则** — 根据已确定的规则，推导出隐含的联动规则（如"订单取消 → 库存回滚 → 优惠券回收"这类链式反应），标注为"[推导]"并请用户确认。

4. **按业务模块分组** — 如果规则跨多个模块，按模块拆分为多张表。

5. **输出格式示例**：

```markdown
## 订单模块 — 业务规则决策矩阵

| 场景条件 | 触发动作 | 异常抛出 | 对应状态变更 | 备注 |
|---------|---------|---------|------------|-----|
| 用户状态为禁用 且 尝试下单 | 阻断操作 | UserDisabledException | 无变更 | |
| 订单金额 > 10000 且 未实名 | 跳转实名认证 | 无，返回 code: 4031 | 订单状态：挂起 | |
| 订单已支付 且 用户发起退款 | 创建退款工单 | 无 | 订单：已支付→退款中 | 需财务审批 |
| [推导] 退款成功 | 回滚库存 + 回收优惠券 | InventoryRollbackException | 订单：退款中→已退款 | 请确认 |
```

6. **产出矩阵后**，告诉用户："这份决策矩阵可以直接用于 Vibe Coding。如果你需要生成 Java 代码骨架，把矩阵给我，我帮你生成策略模式骨架。"

## Step 3: 策略模式骨架

### 目标
基于决策矩阵，生成可扩展的 Spring Boot 策略模式代码骨架。核心设计原则：**新增规则只需加一个实现类，无需改动已有代码**（开闭原则）。

### 输入
- Step 2 产出的决策矩阵（必需）
- 用户指定的包名、模块名（可选，缺失时询问）

### 执行流程

1. **分析决策矩阵**，识别：
   - 校验策略（前置条件检查）
   - 业务动作策略（核心逻辑执行）
   - 状态流转策略（状态机变更）

2. **生成代码骨架**，参考 `references/strategy-pattern-template.md`，包含：

   **a) 策略接口**
   ```java
   public interface ValidationStrategy {
       boolean supports(BusinessContext context);
       ValidationResult validate(BusinessContext context);
       int order(); // 执行优先级
   }
   ```

   **b) 策略工厂** — 使用 Spring 依赖注入自动收集所有实现
   ```java
   @Component
   public class ValidationStrategyFactory {
       private final List<ValidationStrategy> strategies;

       public ValidationStrategyFactory(List<ValidationStrategy> strategies) {
           this.strategies = strategies.stream()
               .sorted(Comparator.comparingInt(ValidationStrategy::order))
               .collect(Collectors.toList());
       }

       public List<ValidationStrategy> getApplicableStrategies(BusinessContext context) {
           return strategies.stream()
               .filter(s -> s.supports(context))
               .collect(Collectors.toList());
       }
   }
   ```

   **c) 具体策略实现** — 每条决策矩阵规则对应一个 `@Component` 类

   **d) 业务上下文对象** — 封装决策矩阵中涉及的所有字段

   **e) 编排入口** — 调用策略工厂 + 顺序执行的 Service 方法

3. **为决策矩阵中的每条规则生成一个具体实现类**，类名以规则含义命名，内部包含完整的 `supports()` 和 `validate()` 逻辑。

4. **输出说明**：
   - 代码中标注 `// TODO` 的地方是需要用户根据实际业务补充的
   - 附带一段"后续迭代话术"，告诉用户新增规则时怎么跟 AI 沟通

### 后续迭代话术模板

当产品补充了新规则时，用户只需告诉 AI：

> "基于之前的决策矩阵，新增一条规则：当 [场景条件] 发生时，执行 [触发动作]，异常抛出 [异常类型]，状态变更为 [目标状态]。请实现一个新的 ValidationStrategy 实现类，并注册到工厂中。"

---

## 关键约束

1. **Step 1 的问题必须是选择题/判断题** — 这是这个 skill 的核心差异化价值。产品经理的时间宝贵，让他们做选择题比写小作文快 10 倍。

2. **只问真正缺失的** — 如果用户已经提供了上下文或背景信息，不要重复追问已有答案的部分。问题数量控制在 10-25 个，上下文越充分，问题越少越精准。

2. **不在 Step 1 写任何代码** — 只输出问题清单。过早涉及技术细节会让产品经理困惑。

3. **Step 2 的决策矩阵必须是 Markdown 表格** — 这是 AI 编码工具理解力最强的格式。

4. **Step 3 必须遵循开闭原则** — 每条规则一个类，新增规则不改已有代码。如果用户的需求规模不适合策略模式（比如只有 2-3 条简单规则），主动建议更轻量的方案。

5. **每步结束后主动引导下一步** — 但不自动执行，等用户明确指示。

6. **推导规则必须标注 [推导] 并请用户确认** — 避免 AI 臆造业务逻辑。
